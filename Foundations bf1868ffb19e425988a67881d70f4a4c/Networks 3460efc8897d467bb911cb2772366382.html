Definitions

TCP - Transmission Control Protocol

  • Provides the standard that machines use to speak to each other
    • This is an agreed upon standard so every machine is able to establish an open two-way connection, until the connection is terminated by one of the machines.
      • used to structure the data and workflow of data over the network.

IP address - Internet Protocol Address

  • the address of the machine - like a phone number

LAN - Local area network

  • all the devices connected to one router

NIC - Network interface card

  • acts as the ear and the mouth of the computer → for talking and listening to other machines

WAN - Wide area NEtwork

  • a network of lans connected to a bigger router → ie an ISP (internet service provider)

MAC Address - is a uniqe identifier for the NIC..

  • This address identifies different clients that are connected to the sam router. This is the last part of the information chain on the client side that directs the information to the correct client
    • represented in hexadecimal format

  1. client sends out a request packet using TCP, containing the following info
    1. who you are: MAC and IP address, port
    1. WHo you want to connect with: MAC and IP address, port
    1. What you want to recieve from that address
  1. The recieving client or server respods with the following:
    1. This is who you are looking for
    1. and you are...
    1. you want this? here you go
  1. other clients can hear this request, however it is ignored since it is not addressed to them it is ignored

Net Module

Allows node apps to use TCP

  • either creating a server or creating a connection
  • all action happens in the callback of that event.

This is how one app on a machine can call up another app on a separate one.

all the client needs is:

  1. the destination IP address [host]
  1. PORT information

SERVER

// import net module
const net = require('net')

// bind the net method, createserver() to a variable
const server = net.createServer();

server.on('connection', client => { // client connection is the argument passed into the callback .. 
  // my computer is the object which the methods are bein called upon, since they are in this cod eblock
  // note that this callback can handle multiple connections or cllients
  console.log('new client connected'); // this .on method is asyncr since itss task is depndent on connection speed
  client.write('hellow there!'); // the callback is executed when the connection is established


  // this is how a server recieves messages from this [arg] or client connection
  client.setEncoding('utf8');
  client.on('data', (data) => { // .on 'data' recieving
    console.log('Client says: ', data);
  });
});
// chain .listen method from the net module -> these are asyncrhonous ?? they are just listening, 
// not initiating a task that will get in the way of other tasks.. 
// but the call back is executed when the listening condition is met.
server.listen(3000, () => console.log('Server listening on port 3000!')); // needs to happen first.. 
// which is fine since it happens before the event loop. Listening and action on listening [with a callback] is not necessarily async
// remembee that is the task that determins async behaviour.. 
// Async is only there to handle tasks that range in time, ie., dependent on connection speed or loading, etc.

// so from the server all of the events pertaining to the client, ie., my computer are happening in the callback.. 
// which has the client scope.. 
// all the methods are applied to the client object.

HTTP

Hyper Text Transfer Protocol

  • HTTP is a protocol used to read and write "resources" (data) in a simple text-based manner.
    • an HTTP request is made up of many components, but right nowypu only have to focus on the PATH and the METHOD
      • the path says what resource the client wants to act on and the method says what action it would like to perform. → verbs

  • HTTP is a request-response protocol, where the client makes requests and the server sends responses
  • HTTP is a text based protocol, making it easy to read and understand for humans
  • HTTP requests must contain the verb/method (eg: GET) and the Path (eg: /about)
  • HTTP requests aren't always to receive data, but sometimes to save data, like when we submit a form on a website. This is done via a POST instead of a GET
  • Requests and responses both contain key-value based headers (eg: Accept-Language: fr, Content-Type: text/html, etc.)

HTTP Methods

  • GET: used to "get" some data from the server
  • POST: usually used to create some new data
  • PUT: generally used for editing existing data on the server
  • DELETE: used to delete some existing data

"Another name for HTTP methods is "verbs", since the methods can be seen as "action words"."

HTTP and TCP

HTTP is not alternative to TCP, but rather the text-based "language" that is used to communicate over a TCP connection. This means that every time our browser accesses a website, it makes an HTTP request by opening a TCP connection on port 80 (or 443 for https) to a given HTTP server's IP address. It then sends text-based commands such as GET /path-to-page HTTP/1.1\r\n along with many other request headers (such as the user agent string, etc.) and waits for a response from the server. The response could be any type of document, which typically is either HTML, CSS, or JS.

HTTP RESPONSES

Status codes

The status code is a three-digit number that the server puts in the response to let the client know whether or not the operation was successful.

ie., some of these include:

  • 200: "Everything went great!"
  • 201: "The request has succeeded and a new resource has been created as a result."
  • 300's: redirection
  • 404: "Resource was not found."
  • 500: "The server had an error."

HEADER

HTTP headers let the client and the server pass additional information with an HTTP request or response. An HTTP header consists of its case-insensitive name followed by a colon (:), then by its value. Whitespace before the value is ignored.

BODY

The response will also usually contain some data, such as the data the client originally requested. This data is stored in a part of the response which is called the body. The body may store data in many kinds of formats, such as text and images. For our purposes, the body will often contain webpages (HTML) or data encoded in JSON, which we'll learn about later on.

Character Encoding

As a content author or developer, you should nowadays always choose the UTF-8 character encoding for your content or data. This Unicode encoding is a good choice because you can use a single character encoding to handle any character you are likely to need. This greatly simplifies things. Using Unicode throughout your system also removes the need to track and convert between various character encodings.
Content authors need to find out how to declare the character encoding used for the document format they are working with.
Note that just declaring a different encoding in your page won't change the bytes; you need to save the text in that encoding too.
As a content author, you need to check what encoding your editor or scripts are saving text in, and how to save text in UTF-8. (It's usually the default these days.) You may also need to check that your server is serving documents with the right HTTP declarations.
Developers need to ensure that the various parts of the system can communicate with each other, understand which character encodings are being used, and support all the necessary encodings and characters. (Ideally, you would use UTF-8 throughout, and be spared this trouble.) -w3.org

DNS record Types

A list of DNS record types

A small subset of common use cases:

  • A: most common; map a hostnames to IP address (IPv4, 32-bit address)
  • NS: Name Server that is responsible for a DNS zone
  • MX: Mail Exchange record specifies where email gets sent to
  • CNAME: Canonical Name, an alias for another hostname
  • AAAA: similar to A, but uses IPv6, 128-bit address

EXAMPLE: Trace a DNS request

$ dig +trace google.com

// or

$ dig @8.8.8.8.8 +trace google.com

// to specify google server.

Servers

The 400-series HTTP status codes are Client Errors – the common ones are valuable to know (namely, 404, 401, and 403).

const util = require('util');
const http = require("http");
const PORT = 3000;
const PATHS = {
  "/pathOne": 'you are in path ONE',
  "/pathTwo": 'you are in path TWO',
  "/": 'welcome stranger',
  "/urls": "www.google.com\nwww.thefriendlystranger.com\nwww.bradpitt.com"
}

// a function when passed as a callback, handles request and response objects created by the createServer() function.
const requestHandler = function(request, response) {


  // the following is how you handle routing to different pages:
  if (request.url === '/urls') {
    response.end(PATHS['/urls'])
  } else if (request.url === '/') {
    response.end(PATHS['/'])
    
  } else {
    response.statusCode = 404;
    response.end(`awww snap, Page Not Found. Status code: ${response.statusCode}`)
  }
  
};

const server = http.createServer(requestHandler);

server.listen(PORT, () => {
  console.log(`Server listening on: http://localhost:${PORT}`);
});

/* 
* you have to handle the status code changes manually as you would any error.. response.statusCode = 404; response.end('404 page not found')

* Routing is leterally a conditional and serving out a different outcoome using the response object -> response.end
