‚ÄúThe state in React allows us to dynamically change one or many elements at once based on one variable. State encompasses the key parts of our UI that change based on user input. State allows us to retain information across multiple renders. To store state, we need to use a feature of React called "hooks" - specifically, the useStatehook.‚Äù -LHL

Import the useState hook

import React, { useState } from "react";

The pair of values returned in the array are related:

  • One is used to get a value (a "getter)
  • The other is used to set a value (a "setter").

function Application(props) {
  const [count, setCount] = useState(0);

  return (
    <main>
      <Button onClick={() => setCount(count + 1)}>Increment</Button>
      <h1>Button was clicked {count} times.</h1>
    </main>
  );
}

// the setter, setCount has to be called in a callback to throw it into async

//NOT: 
//<Button onClick={setCount(count + 1)}>Increment</Button>
// This is wrong! The function would be invoked immediately as soon as the page renders.

the params of useState are positional so choose names that reflect the state you are tracking, using the convention

[state, setState]

is like:

[read, update] = useState(create)

  • where create is the starting value or initial state

modifying the state

use the second value provided by the hook, the first one is a reference to the value in our state, the getter:

let [x, setX] = useState(42);
// WRONG!
x = 99;

// CORRECT!
setX(99);

setState is a function that can change the value and trigger a re-rendering of our components.

to keep JSX clean when performing actions on events is to create a function that only needs to be referenced

const handleClick = (event) => setState(event.target.value ? event.target.value : "");

  <button onClick={handleClick} />

  // instead of

  <button onClick={event => setState(event.target.value ? event.target.value : "")} />

Lifting up the state

From the React docs: In React, sharing state is accomplished by moving it up to the closest common ancestor of the components that need it. This is called * ‚Äúlifting state up‚Äù*.

Passing Down A Way To Change The State

Since the function is declared in App , even if we reference it in the LightSwitchButton component it will run with the context of App . That means that it will be able to change the value of the light state and trigger a re-render.

It is good practice to keep logic that changes the state near the declaration, and not in a child component. The reason behind this is that if the code that changes the state is written in a component that is several levels down, it can be hard to find the source of a bug.

  • keeping the handler close to useState declaration
    • pass down the handler-setter, and the getter-for the conditional render

    Logic surrounding state updates should be declared in the same component as where the state was declared.

Stateful Data

When controlling form input data, it makes sense to have the state locally, since you are not updating the entire state of the app by entering information and monitoring the changes in real time.

The real-time changes to the form are local to the form since you are only entering field data. Once you save your form, then it makes sense to have a more elevated state updated to the new default value.

  • when a user is interacting with the <Form>, the values need to be updated immediately to give the user visual feedback. Once a user inputs data and saves, the <Form> disappears and is replaced by a saved appointment. Outside the view of using the <Form>, we don't need to store the student or interviewer in state because they are not being changed; they are not bound to an interactive element.
  • So, we're choosing to hold the interviewer and student in state in the <Form>. Those are the only pieces of data the <Form> requires to do its job.

Managing State

2 ways to update a state when they are all kept in one state object:

  1. spread Operator
    const state = { day: "Monday", days: [] };
setState({ ...state, day: "Tuesday" });

  1. Object.assign
    const state = { day: "Monday", days: [] };
setState(Object.assign({}, state, { day: "Tuesday" });

Immutable Update Patterns using Spread {...}

ie., Copies only one level of keys

cont original = { one: 1 };
const bad = original;
const good = { ... original };

console.log(orginal === original); // true
console.log(orginal === bad); // true
console.log(orginal === good); // false

ie., How to copy multiple level of keys

const original - { one: 1, sub: { two: 2 } };
const copy - { ...original, sub: { ...original.sub } };

console.log(original === copy); // false
console.log(original.sub === copy.sub); // false

Note that it is best to avoid this kind of complexity. Keep state nesting to a minimum. The work around for that is to keep your data structured in a normalized form. Using id‚Äôs and then using selector helper functions to append the correct information when you are retrieving said data.

Merging multiple Objects

const first = { one: 1 };
const second = { two: 2 };
const copy = { ...first, ...second };

console.log(copy === copy); // true
console.log(first === copy); // false
console.log(second === copy); // false

/* first */
{
  one: 1
}

/* second */
{
  two: 2
}

/* copy */
{
  one: 1,
  two: 2
}

Aliasing Actions

Even though all of the states are combined into a single object, we can still have separate actions to update certain parts of the state by creating a function that acts on setState, but with the spread operator and a param to change, observe:

const setDay = day => setState({...state, day});

dependency Error

if you find that you get an error:

The issue is that we are now referring to state in the effect method, but we haven‚Äôt declared it in the dependency list. In fact: we don‚Äôt want to.

const setDays = days => setState({...state, days})

So to remove this dependency, we must pass setState a function, so the dependency is not detected in the useEffect

const setDays = days => setState(prev => ({...prev, days}))
