EXAMPLE:

The <Appointment> component is complex enough that we split it into many smaller components. Each of these components has a responsibility.

  • Header displays the time for the appointment
  • Empty allows a user to choose which time slot to book
  • Show allows a user to see an existing appointment
  • Confirm allows a user to confirm a destructive action
  • Status informs the user that an operation is in progress
  • Error informs the user when an error occurs

Building all of this functionality into one component is hard to manage. The stories that we wrote allow us to test each of the smaller components in isolation. Alternatively, we would have to build out all of the <Appointment> logic to be able to test each of the views. We won't be adding error handling to our application immediately, but when we do we will have our component ready for it.

Fragments

import React, { Fragment } from 'react';

Use fragments when you do not want to add unnecessary DOM nodes (i.e. extra <div> tags). Use shorthand where possible, but use <Fragment> if the component should be passed props.

return (
  <React.Fragment>
    <h1>A heading</h1>
    <p>A paragraph</p>
  </React.Fragment>
)

// or

return (
  <Fragment>
    <h1>A heading</h1>
    <p>A paragraph</p>
  </Fragment>
)

// OR


return (
  <>
    <h1>A heading</h1>
    <p>A paragraph</p>
  </>
)

One of the rules of React is that JSX expressions must have only one parent element.

This means that:

  1. Components can only return JSX with one root element.
  1. Conditional expressions that resolve to JSX must also only have one root element.

Unlike a <div> , a React <Fragment> does not create any extra DOM nodes, which can clutter up your HTML output and cause problems with your layout.

.. having too many unneccessary div tags can result in CSS collissions.

Conditional Renders with multiple tags must resolve to a fragment in the ternary or short circuit operation. even though that, itself is in a fragment.

return (

    {userLoggedIn ?

    </>
    }
