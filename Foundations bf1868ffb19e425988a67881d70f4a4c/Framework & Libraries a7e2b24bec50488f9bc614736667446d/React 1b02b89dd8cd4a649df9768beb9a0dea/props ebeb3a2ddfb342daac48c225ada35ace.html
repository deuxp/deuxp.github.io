using props

The properties ‚Äúprops‚Äù you pass to a component are automatically stored in an obj. That obj is available as a parameter in the component definition and is commonly named props.

ie.,

function Profile(props) {
  // props will contain an object: {firstName: "Amy", lastName: "Mansel", avatar: "/profile-hex.png"}
  const firstName = props.firstName;
  const lastName = props.lastName;
  const avatar = props.avatar;

  return (
    <aside>
      <div className="profile">
        <img className="profile__image" src={avatar} />
      </div>
      <br />
      <div className="profile__name">
        <h2>
          <span className="profile--bold">{firstName}</span> {lastName}
        </h2>
      </div>
    </aside>
  );
}

<Profile doStuff={doStuff} />;

props.children

The props object in React has a built-in children key. props.children. The contents are. anything that is inside the tags of a component.

The point to take away from this is that props.children is anything that is passed to a component inside its tags. Within any component, we can access the children through props.children. A lot of libraries and some more advanced patterns take advantage of this concept.

ie., 

// dont worry about the style={{}} for now
// focus on what is inbetween the tags

function WelcomeMessageBox(props) {
  return (
    <div
      style={{
        height: "30%",
        width: "70%",
        marginLeft: "auto",
        marginRight: "auto",

        backgroundColor: "teal",

        padding: "10%",
        fontSize: "2em",
        fontFamily: "Futura",
      }}
    >
      {props.children}
    </div>
  );
}

naming Props for Event Handlers

Lastly, we want to take a moment to point out a naming convention that may cause some confusion as to how event handlers in React work. It has to do with passing functions to components that contain event handlers.

Consider two components, <Parent> and <Child>, where the parent passes down a function as a prop to the child:

The one in the Component is aprop being passed down, ,whereas the child it it is being uesd in a React.createElemetnt() essentially - of a div element so when it is used there it is JSX being translated by Babel into a DOM element. So it is interpreting the onCLick as an event handler in that case.

function Parent(){

  function doSomething() {
    console.log("Do something");
  }

    return (
      <h1>I am the parent. Here is the child.</h1>
      <Child onClick={doSomething} />
    );
  }

onClick in the <div> tag in the second code block is a keyword. It is used to define an event handler and it has to be named onClick to successfully listen for a click event. However, onClick in the first code block as the name of a prop being passed to <Child> is not a keyword! It is just a regular prop name with no special significance. It could be named anything, but we named the prop onClick so that it's easy to to see why it is being passed to the <Child> .

function Child(props) {
  return <div onClick={props.onClick}>Click me!</div>;
}

Prop drilling

The process of getting data down to the components that need them, lower on the React Component tree. Also known as ‚Äòthreading‚Äô

Being able to pass props down through a component hierarchy is a critical skill for any React developer. It's important to remember that there is no perfect approach to computing props and creating actions. As a rule, we should try and reduce the amount of duplicated state within the application.
